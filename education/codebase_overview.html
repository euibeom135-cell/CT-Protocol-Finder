01 The Big Picture

You describe a first-in-human study; the tool returns the most relevant dose-escalation protocols, ranked by clinical pharmacology value.

You describe the first-in-human study you're planning — drug modality, therapeutic area, study design. The tool searches ClinicalTrials.gov for comparable dose-escalation protocols, scores each one for clinical pharmacology relevance (FIH/SAD/MAD designs, PK-rich endpoints, and appropriate modality are weighted highest), and returns a ranked list with the most data-rich protocols on top.

You can preview any protocol — the tool extracts structured sections and tables from the PDF and produces a CP-focused assessment (drug, design, PK elements, dosing, enrollment). A separate similarity engine finds trials with comparable designs across seven FIH-specific dimensions: PK question type, drug modality, escalation method, population, route, DLT window, and PD biomarker readouts.

02 How Trial-Design Similarity Is Measured

When you search for FIH trials with similar designs, the engine compares every candidate against your query across four independent questions and merges them into one number. Here is exactly how that works, using a real pair of trials.

Scope: This is a trial-design similarity engine. It measures how similar two trials are as studies (escalation method, population, route, etc.). It does not measure whether two drugs are pharmacologically similar. The axes a clinical pharmacology reviewer needs for FIH comparator selection — tox profile, ADME, target/mechanism of action, drug-drug interactions, dose-selection basis — are not in this engine. A separate drug-pharmacology similarity layer is planned.

03 How the AI Finds FIH Protocols

Seven steps from your FIH question to ranked dose-escalation protocols. Some steps are deterministic rules (same input, same output every time). Others use AI judgment that you should spot-check.

Scripted — Deterministic rules. Same input always gives same output.
AI judgment — LLM reads and reasons. Spot-check these steps.
1

Understand your FIH question

The LLM reads your natural-language query and extracts structured fields: therapeutic area, drug, phase, and FIH study design type (SAD, MAD, food effect, DDI, etc.). If your question is ambiguous, it asks a clarifying question first.

2

Search ClinicalTrials.gov

Three parallel queries hit the registry with different search strategies (broad, synonym-expanded, and tight) to maximize recall. Results are merged and deduplicated.

3

Filter to CP-relevant trials

A rule-based scoring system (0–100) rates each trial for clinical pharmacology relevance. Vaccines, devices, behavioral studies, and topicals are filtered out. FIH/SAD/MAD designs and systemically absorbed drugs score highest. Only trials above the relevance threshold survive.

4

Rank by match to your question

The LLM reads each surviving trial and scores 0–100 for how well it matches your original query. This catches nuances the rule-based filter misses.

5

Boost and sort

Fixed-math boosts are applied (study-type bonus +15, CP score bonus +10) and the final list is sorted by combined score. No AI judgment here — just arithmetic.

6

CP advisory (optional)

If you request it, a more capable LLM reads the top 3–5 trials and writes an FIH-focused recommendation: why these protocols are relevant to your dose-escalation question, what PK data to expect, and what design features matter for your starting-dose rationale.

7

Protocol preview

When you click "Preview" on a trial, the system downloads the protocol PDF, extracts structured content (tables, sections, headings), and the LLM produces an FIH-focused assessment: drug and mechanism, study design (SAD/MAD/crossover), PK sampling strategy, dose levels, escalation rule, and enrollment.

Trust pattern: Every AI step outputs structured JSON before the next step runs. You can see exactly what the LLM decided and fix it before any database query runs. The scripted steps are fully deterministic — if you feed them the same input twice, you get the same output twice.

Architecture Map

Four figures built from the actual source code. Figure A shows the static module dependency graph (who imports whom). Figures B–D show the three sequential execution flows (search, preview, similarity) with real function calls at each step.

Figure A. System architecture and module dependencies. Solid violet arrows denote Python import relationships. The FastAPI entry point app.py imports 7 service modules (Layer 3); similarity_engine composes ontology_engine (Layer 4), which in turn uses three ontology modules (Layer 5). A non-obvious dashed rose arrow shows that fda_client imports NON_CP_KEYWORDS from cp_agent, coupling the FDA client to CP domain logic. All service modules share Pydantic data classes from models.py (dashed indigo). Dashed yellow arrows terminate at external APIs and local libraries.
Figure B. Search execution flow. Sequential trace of a single user query from natural-language input to ranked trial list. Step 6 shows the three parallel search strategies (broad / synonym-expanded / tight) that give the system high recall. The pipeline combines one LLM call for intent parsing (step 2), one rule-based CP filter (step 9), and one LLM re-rank (step 10), then two numerical boosts (step 11). Step 14 is an optional follow-up route (/api/cp-advise) that uses the more expensive gpt-4o model for expert recommendations.
Figure C. Protocol preview flow. Triggered when the user clicks "Preview" on a trial card. The pipeline hits three network endpoints sequentially: CT.gov CDN (step 2), docling (steps 3–4; note the pdfplumber fallback at 4b), and OpenAI (step 7). The critical handoff is step 6, where format_sections_for_llm serializes the parsed section dictionary into a bounded prompt so the LLM can produce a structured 7-field CP assessment.
Figure D. Trial-design similarity search flow (two variants merging). Both entry points converge at step 6 (FAISS oversampled search). Step 7 is the heart of the system — ontology_engine.rerank_results performs three independent similarity computations per candidate (MeSH conditions, drug classes, FIH taxonomy) and combines them with the FAISS embedding score using fixed weights 0.25 / 0.30 / 0.30 / 0.15. The FAISS step oversamples by 3× so that ontology-strong but embedding-weak candidates can still surface. Step 8 attaches a CP relevance score so results are ranked by both design similarity AND CP value.

Scoping note: this engine answers "what other FIH trials have similar designs" — not "what drugs are pharmacologically similar enough to inform my new molecule's FIH." The latter question (targeting tox profile, ADME, MoA, DDI, dose-selection basis) is the job of a separate drug-pharmacology similarity layer that will be built from FDA Clinical Pharmacology review PDFs.
